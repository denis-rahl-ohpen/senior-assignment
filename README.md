# Senior Frontend / BFF Engineer — Take-Home Assignment

## Case Readiness Dashboard

**Time expectation:** ~6 hours. This is intentionally not completable to "perfect" in that time — we'd rather see strong judgement on part of the problem than a rushed pass at all of it.

---

## 1. Context

You're joining the team that builds our mortgage mid-office platform — the internal tool mortgage officers and underwriters use to process applications after submission and before a final loan decision. In production, this platform integrates with several domain services: applicants, documents, affordability, risk, valuation, decisioning, audit, and case workflow.

For this assignment we've cut a single, realistic vertical slice out of that platform: a dashboard that helps an underwriter quickly see whether a case is ready to move to a decision, and why (or why not).

## 2. Objective

Build a small full-stack slice consisting of:

- a **BFF (backend-for-frontend)** layer that aggregates data from several simulated upstream services into a single shape for the frontend, and applies a "case readiness" business rule
- a **frontend** (case list + case detail) that presents this clearly to an underwriter
- **one state-changing action** the underwriter can take from the detail view

You're free to choose your own stack for the BFF (Node/Express, Next.js API routes, etc.) and frontend (React is preferred given the team's stack, but use what best shows your strengths). No specific framework is mandated beyond that.

## 3. What's provided

A `mock-data/` folder containing static JSON representing five simulated upstream services, for 8 sample cases:

| File | Simulates | Key fields |
|---|---|---|
| `cases.json` | Case service | id, product type, requested loan amount, stage, applicant refs |
| `applicants.json` | Applicant service | id, name, role, income, employment type |
| `documents.json` | Document service | per-case list of required documents and their status |
| `affordability.json` | Affordability service | DTI ratio, verdict |
| `risk.json` | Risk service | risk score, flags (type, severity, status) |
| `valuation.json` | Valuation service | property value, LTV, valuation status |

How you consume this data is up to you — load it directly in your BFF, serve it via a tiny local JSON server, or wrap it in fixtures. There's no need to build real upstream microservices; the point is the aggregation and rules layer on top of them.

One thing worth noting up front: **one case's valuation record represents a failed upstream call rather than normal data.** How your BFF and UI handle that is part of the exercise, not a bug in the data.

## 4. Business rules — case readiness

A case is generally considered **ready for underwriter decision** when:

- all mandatory documents have been received and verified
- the affordability assessment did not fail
- there are no unresolved risk flags
- valuation is complete and the loan-to-value (LTV) ratio is within policy (max **90%** for this exercise)

We've deliberately left parts of this underspecified:

- what exactly counts as "unresolved" for a risk flag (does severity matter? does an acknowledged-but-not-resolved flag count?)
- how to treat an affordability verdict of `MANUAL_REVIEW`, which is neither a clean pass nor a fail
- what "within policy" means at the boundary (is exactly 90% LTV acceptable?)
- what a case's readiness status should show when an upstream service fails to respond

We're not looking for the one "correct" interpretation. We want to see you notice the ambiguity, make a reasonable call, and explain it — the same way you would with a product manager who hasn't specified every edge case.

## 5. Functional requirements

**5.1 Case list view**
A list of the 8 cases showing, at minimum: case id, primary applicant, product type, loan amount, and a readiness indicator (e.g. Ready / Not Ready / Needs Attention — naming is your call).

**5.2 Case detail view**
Selecting a case shows applicant info, document statuses, affordability summary, risk flags, valuation/LTV, and a readiness panel that explains *why* the case is or isn't ready (not just a yes/no badge).

**5.3 One state-changing action**
Implement at least one of:
- **Request missing document** — for a document in `MISSING` status, let the underwriter trigger a request; reflect the updated status in the UI.
- **Acknowledge a risk flag** — for an `OPEN` flag, let the underwriter acknowledge it with a short note; reflect this in the UI and in the case's readiness explanation.

No real persistence is required — in-memory state on the BFF (reset on restart) is fine. If you want to go further, a simple audit log entry generated by this action is a nice touch, but it's optional.
